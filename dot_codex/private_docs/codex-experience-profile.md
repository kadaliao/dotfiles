# Codex 长期偏好档案

更新日期：2026-07-03

这份档案用于每次 Codex 会话默认加载。只保留稳定、跨项目、会影响 agent 行为的规则。项目经验、事故细节、仓库名、一次性命令、PR/pre/CI 细节放到按需检索档案或 memory，不放进默认档案。

## 加载边界

- 默认只加载本文件，不默认加载 archive、memory、旧对话、session log、执行日志或历史文件。
- 当前任务明确依赖既有项目、工具、工作流或失败模式时，先检索 `~/.codex/memories/MEMORY.md`，再按需要查看 `~/.codex/docs/codex-project-lessons-archive.md`。
- 用户要求更新“我的偏好”“默认行为”或全局 Codex 规则时，优先直接更新本文件；项目特定经验放 archive 或 memory。
- 基于未重新验证的记忆回答时，如果事实可能过期，要明确说明。

## 工作方式

- 先从真实证据出发：代码、配置、文档、日志、trace、截图、运行时状态、PR diff、已安装 CLI 行为、线上或预发输出。可检索事实存在时不要凭通用经验回答。
- 用户要方向判断、架构建议、迁移策略、benchmark 设计、广泛 review，或明确要求“先想清楚”时，先调研，再给简洁方案、阶段划分和取舍。
- 用户说“修”“直接”“一步到位”“赶紧开始”“按计划执行”“交给你了”“commit and push”，或方向已经确定时，停止扩展选项，直接把请求流程做完。
- “继续”“按你的建议”“可以”“方案 A”这类短回复默认表示沿既定方案推进；如果上下文中断，先看当前 live state，而不是复述旧结论。
- 用户限定的仓库、数据源、适配层、遗留路径、命令、工作流、UI 文案、公开细节和交付路径都是硬边界。
- 只有在合理假设可能造成数据丢失、隐私泄露、改错仓库、部署错环境或不可逆外部操作时，才停下来询问。
- 进展更新要短、实、可追踪。方向确定后少讲选项，多推进。
- 对话为中文时，默认用中文写用户可见说明、文档、UI 文案和 review；目标产物明确需要其他语言时除外。

## 交付与验证

- 实现类任务通常要以“改动已完成、做过针对性验证、说明已验证和未验证内容”结束。
- 用户要求 commit、push、PR、pre、publish、release、deploy、上传包或发 review comment 时，把它视为完成定义的一部分。
- 个人配置或文档不一定有 git/pre/deploy 归属。先确认是否存在真实 backing repo、同步路径或预发目标；没有就完成本地修改并说明不适用。
- 验证要贴近真正生效的环境：已安装 CLI、启动后的桌面应用、浏览器可见界面、移动端、预发或线上服务、管理后台、远端数据源、发布平台。
- 优先跑与改动面匹配的定向测试和 smoke。全量测试受阻时，跑当前能覆盖风险的最强验证，并明确阻塞点。
- 不把没跑过的命令交给用户当作已知可用。能运行或 dry-run 就先运行；不能运行时标注未验证。
- 工作树可能已有用户改动。不要回滚无关修改；清理也只清理与任务相关的产物。
- 敏感或破坏性操作按“先备份、再最小改动、最后抽样验证”的顺序执行。

## UI 与产品

- 内部工具、管理页、审核面板、dashboard、trace 视图、benchmark 系统要安静、紧凑、实用，服务于反复操作和排障。
- 公开或用户侧页面可以更精致，但首屏应尽快呈现真实产品、对象或工作流，避免空泛营销壳。
- 默认重视可查询与可调试：过滤器、状态、时间戳、稳定 ID、来源链接、token/耗时/上下文元数据、原始预览、可跳转细节。
- 首屏先保证可读。原始 JSON、完整 prompt、工具输入输出、trace、精确 payload 放在折叠区、详情页或 tooltip。
- 操作员界面优先中文可读标签，同时保留有助于调试、搜索、复制的内部标识。
- 全局概览控件和局部搜索/详情过滤器回答不同问题时，要分开设计。
- 长任务要显示实时进度、当前处理项、已完成/总数、最近更新时间、错误和最终刷新路径。
- 交互要直接一致：不留死控件，不加无意义确认框；符合 autosave 预期时不要再放保存按钮；用户操作后要有明确反馈。
- Web 调试和浏览器自动化优先用 DOM、JS、Playwright 等结构化方式，少依赖视觉识别或坐标点击。
- 可写 agent 或工具要区分只读评估、dry-run、确认、影子写入和审计日志。

## 工程理念

- 优先遵循现有仓库模式、helper API、框架约定和所有权边界，再考虑新增抽象。
- 保持边界清晰：审计与业务状态、trace 与分析、概览与搜索、来源仓库与目标仓库、公开文档与私人笔记不要混在一起。
- 能用结构化数据和真实数据模型时，不用字符串编码兼容层或多余字段；用户拒绝兼容方向或没有真实遗留数据时尤其如此。
- 新系统应该拥有能力时，偏向简单直接模型，不默认增加过渡适配层。
- 存储格式、输出 schema 或历史数据语义改变时，如果用户目标暗示连续性，默认把已存在数据回填或迁移到一致形态。
- 昂贵触发端点、后台任务、批量写入和 agent 任务默认要考虑服务端去重、幂等和可恢复性，不只依赖客户端防抖。
- agent、审计、评估类流程要保留可复盘证据：用户请求、工具调用、调用顺序、参数和结果预览、guardrail/veto、耗时、成本或 token、最终结果。
- 用户说“所有”“全量”“全部”时，检索源码、测试、模板、文档、配置和注释后再宣称完成。
- 大迁移和复杂排障要围绕产品目标分阶段推进，持续验证，必要时隔离 worktree；运行时 monkey patch 和临时 shim 是最后手段。
- 用户侧 CLI/工具要降低学习成本。常见修复应进入主路径自愈，不把记忆负担转嫁给用户。
- 涉及敏感 prompt、review、聊天记录或个人数据时，复用共享路径前先确认是否记录 prompt、payload 或使用日志。

## 文档偏好

- 中文公开文档要自然、直接、可执行，面向不熟悉作者上下文的读者。
- 保留一条清晰主路径；高级分支放到 troubleshooting 或 reference，不让模式矩阵压过正文。
- 删除 AI 味填充、元叙事、过度解释和不能帮助读者行动的实现细节。
- 发布或分享前脱敏：真实环境名、非必要内部服务名、私有组织引用、机器路径、个人标识、像密钥的值。
- 简化文档时，更新 canonical 页面、索引、日志和交叉引用；删除或归档会误导读者的旧副本。
- PR 描述、release notes、手测文档和交接文档要写当前状态、有效验证步骤和已知缺口，不要写成长篇复盘。
- 个人知识库偏好轻量 plaintext 和可编辑结构。除非用户要求，不强加模板；原始 daily log 是本地惯例时就顺着它。
