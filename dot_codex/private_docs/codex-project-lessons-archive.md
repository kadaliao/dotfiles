# Codex 项目经验按需档案

归档日期：2026-07-03

这份文件不默认加载。只有当前任务明确匹配下面的项目类型、工具链、工作流或失败模式时，才把它作为检索路由和经验提示。跨项目长期偏好放在 `codex-experience-profile.md`；具体证据、命令输出、PR/pre/CI 细节保留在 `~/.codex/memories/`。

## 检索规则

- 先加载 compact profile；本档案只在任务触发条件匹配后使用。
- 本档案只提供路由和已知风险提示；行动前仍要读取当前仓库、配置、日志和运行状态。
- 需要精确命令、提交、PR、pre、CI 输出或历史报错时，检索 `~/.codex/memories/MEMORY.md` 和对应 rollout summary。
- 对线上状态、凭据、CI、部署、发布平台和外部 API，把本档案视为可能过期。

## Codex 本地状态与偏好档案

触发条件：
- Codex 历史可见性、桌面/移动线程列表、provider/profile 切换、官方与 relay 配置差异、远程控制状态、app-server 状态、本地 Codex 配置、全局 profile/archive/AGENTS 维护。

可复用经验：
- 偏好档案维护要直接改目标文档，不只在聊天里总结；compact profile 保持薄，项目经验进 archive 或 memory。
- 历史可见性可能同时依赖 session 文件、history/index、SQLite 状态、provider/model 身份、活跃 profile、远程主机选择、preview/title 元数据。
- 修复前先诊断完整链路，再同步修复影响可见状态的来源；只改单个文件经常会让 UI 继续显示旧状态。
- profile 切换工具应把常规修复放进主路径自愈，而不是要求用户记住额外 repair 命令。
- 桌面 app 和 app-server 可能受 launchd 或应用托管环境影响，不一定继承交互 shell 环境。
- Codex 自知识问题优先检查本地配置、文档、已安装 CLI 行为和官方资料，不靠选项名推断语义。

## 内部服务、Git 与预发部署

触发条件：
- 内部服务仓库、内部 GitHub/GHE、PR、预发更新、部署、远程脚本、日志检索、Python 迁移、异步 worker、RPC/prefetch、内部管理路由修复。

可复用经验：
- 用户要求交付时，常见完成链路是：定向验证、commit、push、按需创建或更新 PR、按需更新预发/部署、smoke 验证、清理临时产物。
- 某些仓库有特殊交付边界，例如默认不走 PR、只做本地提交、或必须把产品状态写回 GitHub。任务命中具体仓库时先查 memory。
- 本地测试可能弱于预发或运行时验证。优先组合定向本地测试、应用上下文 smoke、日志和真实路由检查。
- Python/runtime 迁移不能只验证 import 成功；要用下游服务或实际业务面证据确认行为，并处理开发态与预发/线上差异。
- 异步、worker、消息队列相关改动要跑到真实异步路径、worker 路径或消息路径；只测只读 HTTP 通常不够。
- 大版本兼容升级优先从真实失败、依赖约束和运行环境盘点开始；能删除临时 shim 就删除，不能删除时说明阻塞依赖和后续路径。
- 内部 Git 与部署命令常有仓库级路由和权限差异。行动前重新确认 remote、默认分支、worktree 状态和可用部署目标。

## PR、Review 与 CI 跟进

触发条件：
- 用户要求 review、复审、贴评论、处理 reviewer feedback、修 CI、创建/更新 PR、请求 reviewer。

可复用经验：
- review 默认是找 bug、风险、行为回归和缺测试；如果用户语境指向远端 PR，要把 actionable feedback 发到对应 PR，而不是只在聊天里总结。
- 用户说“有更新了”“继续 review”时，重新拉取最新 head/diff，避免重复旧 finding。
- CI 失败要读具体 artifact/log/check 输出后修；不要把本地猜测当作 CI 事实。
- PR 描述要同步当前验证、预发地址、已知缺口和阻塞原因；过期的 pre 或环境信息要更新。

## 管理、审计、Benchmark 与 Agent UX

触发条件：
- 管理页、审核 dashboard、审计日志、trace 视图、benchmark/evaluation 系统、agent 工具调用可见性、token 或成本统计、操作员工作流。

可复用经验：
- 审计、token、trace、benchmark 证据需要独立生命周期和存储边界；不要随手混入附近业务表。
- 为调试和复盘保留足够过程证据：原始用户请求、工具调用、调用顺序、参数和结果预览、guardrail/veto、耗时、token/成本、最终输出。
- 实时 benchmark 或状态页要把轻量轮询状态和昂贵详情分开；`ok: true` 这类原始状态不是分析结论。
- 操作员 UI 先展示中文可读标签和人类友好数值，原始 payload、ID、JSON 按需展开。
- 全局概览过滤器和局部详情/搜索过滤器要分离。隐藏稳定 ID 加可搜索标签，能处理点击值不在当前选项列表里的情况。
- 隐私或合规清单类结果如果用户要“直接结果”，优先表格化输出，避免把代码路径和内部实现细节压过结论。

## 公开文档、包与工具

触发条件：
- 公开 README/docs、包发布、可复用 CLI/工具、外部教程、skill/plugin 文档、对外示例。

可复用经验：
- 公开文档写给外部读者，不写成本机或作者笔记。移除个人路径、私有名称、真实环境名、内部服务细节和像密钥的值。
- 文档过复杂时，优先收窄主路径，而不是保留所有分支。高级内容放 troubleshooting 或 reference。
- 示例要跟当前受支持的包管理和工具链一致；本地工具链变化后要刷新示例。
- 可复用工具应把常规 repair/self-healing 放进主命令流程，诊断命令留给维护者。
- 功能完成后，更新 README/status 文档，删除或归档已经误导的旧文档。

## 原生应用与发布

触发条件：
- iOS/macOS 应用、设备安装和启动检查、发布平台、截图、release notes、产品路线状态、应用内交互打磨。

可复用经验：
- 验证用户实际看到的表面：已安装应用、模拟器/真机行为、截图集、发布元数据、商店状态、购买或内容边界。
- 产品路线状态适合落在项目 canonical tracker 或仓库产物里，不只留在聊天记录。
- UI 打磨要覆盖流程完成后的行为、返回路径、保存状态、空状态和多语言文案。
- 发布工具受账号或凭据限制时，只报告已验证状态和阻塞点，不声称发布已完成。

## 数据、Notebook 与分析

触发条件：
- Spark/Jupyter/notebook、CSV 或数据导出、分析报告、图表、SQL 或数据对账、队列/回填/数据链路排障。

可复用经验：
- 改分析逻辑前先看真实 schema、样本行和实际查询/运行路径。
- 存储输出格式变化后，如果用户目标是连续语义，默认列出现有 corpus、做回填或迁移，并验证业务侧剩余数据，而不只看队列 drain。
- 输出路径和交付物要明确；预览或截断字段足够时，不倾倒完整原始数据。
- 用户要求可视化分析时，产出具体文件，并说明数据切片、假设和残余不确定性。
- 用户纠正业务或数据定义时，更新模型和文档，不围绕错误假设增加兼容层。

## 个人知识、邮件与自动化

触发条件：
- 个人知识库、邮件、日历、提醒、邮箱回溯、个人资料整理、定时简报、外部发送或删除。

可复用经验：
- 汇总个人数据前说明读取范围。默认先覆盖足够广的范围，再按风险、截止时间、变化和需要审批程度排序。
- 需要确认的写操作必须拿到明确确认；模糊短回复不能当作发送、删除、发布或修改外部状态的许可。
- 除非用户要求严格 schema，否则保留轻量 plaintext 和本地知识库惯例。
- 敏感笔记和邮箱流程默认最小暴露：不要倾倒完整原文；需要追踪时保留 source ID 或 hash；清理可能保存明文的临时备份。
- 文章进入知识库时，如本地结构区分 raw capture 和维护后的 synthesis/navigation，两者都要保留。

## 浏览器与本机操作

触发条件：
- 浏览器自动化、本机快捷键或输入问题、桌面应用自动化、已安装 CLI 维护、磁盘清理、SSH/Tailscale/Clash/FRP 等本机网络状态修复。

可复用经验：
- Web 任务优先用 DOM、JS、浏览器协议自动化；视觉点击只作为后备。
- 本机即时故障先做快速具体诊断和可能原因隔离，少做大而泛的理论分析。
- 维护已安装工具时，验证真实 executable 路径、安装版本、包管理来源和升级后行为，不只看源码 checkout。
- 网络/远程登录问题先复现失败路径，再比较别名、解析、路由、认证、host key、代理和服务端监听状态。
- 破坏性清理或涉及持久状态的修复要先备份，只在获准类别内操作，之后抽样验证。
